# C# 语言特性 AOP Emit 适配分析

> 目标：梳理从 C# 6 到最新版本（C# 13）引入的语言特性，分析哪些在 AOP Emit 过程中需要 AspectCore 适配，并按优先级给出适配方案。

## 1. 背景

AspectCore 拥有两套 AOP 引擎：

| 引擎 | IL 生成方式 | 关键文件 |
|------|------------|---------|
| **DynamicProxy**（运行时） | `System.Reflection.Emit` + AST 架构 | `ILEmitVisitor.cs`、`ClassProxyBuilder.cs`、`MethodBodyFactory.cs` |
| **Source Generator**（编译时） | Roslyn 生成 C# 源码字符串 | `ProxyEmitter.cs`、`AspectCoreProxyGenerator.cs` |

当前状态：

- `LangVersion` 设为 `10.0`（核心库），Source Generator 使用 `latest`
- 支持 `net6.0` ~ `net9.0` + `netstandard2.0/2.1`
- 两套引擎在方法体生成上有不同路径：DynamicProxy 通过 `ReturnKind` 枚举分发到 `AspectActivator.Invoke*` 系列方法；Source Generator 根据同步/异步生成不同的内联代码

### 1.1 AOP Emit 核心流程

运行时 DynamicProxy 的 Emit 过程：

1. **构建阶段**：`ClassProxyBuilder` / `InterfaceProxyBuilder` / `InterfaceImplBuilder` 构造 AST（`ProxyTypeNode`）
2. **Emit 阶段**：`ILEmitVisitor` 遍历 AST，通过 `ILGenerator` 发射 IL 操作码
3. **方法体决策**：`MethodBodyFactory.DecideBody()` 选择方法体类型（直接委托、反射器委托、切面激活器、桩）

编译时 Source Generator 的 Emit 过程：

1. **发现阶段**：`AspectCoreProxyGenerator` 发现带 `[AspectCoreGenerateProxy]` 的类型
2. **生成阶段**：`ProxyEmitter` 生成 C# 源码（字符串拼接），由 C# 编译器编译

## 3. 需要适配的特性

### 3.1 P0 — 直接导致代理生成失败或运行时错误

#### 3.1.1 异步流（IAsyncEnumerable\<T\> / IAsyncDisposable）— C# 8.0

**问题描述**

`ReturnKind` 枚举只有 6 种值（`Void`、`Sync`、`Task`、`TaskOfT`、`ValueTask`、`ValueTaskOfT`），没有覆盖 `IAsyncEnumerable<T>` 和 `IAsyncDisposable`。代理异步流方法时，返回值处理会落入 `Sync` 路径或直接失败，导致 `yield return` 语义完全破坏。

**影响范围**

- `AspectActivator` 缺少 `InvokeAsyncEnumerable<T>` 方法
- `ILEmitVisitor.EmitReturnValue` 无对应分支
- `ProxyEmitter.EmitProxyInvokeBody` 无异步流代码生成路径
- 拦截器链不支持 `IAsyncEnumerable<T>` 的延迟执行语义（每次 `MoveNextAsync` 都需执行拦截器链）

**适配方案**

1. 新增 `ReturnKind.AsyncEnumerable` 和 `ReturnKind.AsyncDisposable` 枚举值
2. 在 `AspectActivator` 中新增 `InvokeAsyncEnumerable<T>` 方法，使用 `AsyncIteratorMethodBuilder` + 状态机
3. `ILEmitVisitor.EmitReturnValue` 新增对应分支，发射 `IAsyncEnumerable<T>` 状态机 IL
4. `ProxyEmitter.EmitProxyInvokeBody` 新增异步流代码生成路径，生成 `yield return` 代理逻辑
5. 拦截器链需要支持 `IAsyncEnumerable<T>` 的延迟执行语义

**适配难度**：⭐⭐⭐⭐

---

#### 3.1.2 init-only 属性设置器 — C# 9.0

**问题描述**

两个引擎都生成普通 `set` 访问器，不区分 `init` 和 `set`。当代理类重写 `init` 属性时：

1. 语义错误：允许运行时修改 init-only 属性
2. 协变返回场景下签名不匹配：`init` 访问器与 `set` 访问器在 override 时不兼容
3. IL 层面 `init` 访问器带有 `IsInitOnly` 标志，代理未保持

**影响范围**

- `ILEmitVisitor.VisitProperty` 无 `init` 访问器处理（`ILEmitVisitor.cs:285-311`）
- `ProxyEmitter` 生成 `set` 而非 `init`（`ProxyEmitter.cs:596-625`）
- `PropertyNode` 无 `IsInitOnly` 字段

**适配方案**

1. `PropertyNode` 增加 `IsInitOnly` 字段
2. `ILEmitVisitor.VisitProperty` 中对 `init` 访问器设置 `MethodImplAttributes.InitOnly`（通过 `SetImplementationFlags`）
3. `ProxyEmitter` 中生成 `init` 关键字而非 `set`
4. 构造函数中需要在 `init` 调用前正确加载 `this`（IL 层面 `init` 要求 `this` 在栈上）
5. 协变返回场景需保持 `init` 签名一致性

**适配难度**：⭐⭐⭐

---

#### 3.1.3 required 成员 — C# 11.0

**问题描述**

完全没有处理。代理构造函数不转发 `[RequiredMember]` 属性，也不处理 `SetsRequiredMembers`。当实现类有 `required` 属性时：

1. 代理构造函数缺少 `[SetsRequiredMembers]`，编译器报错
2. `required` 属性的 `set` 访问器可能需要是 `init`
3. 对象初始化器中无法设置 `required` 成员

**影响范围**

- `ConstructorNode` 无 `SetsRequiredMembers` 标志
- `ClassProxyBuilder.BuildConstructors` 不转发 `[SetsRequiredMembers]`（`ClassProxyBuilder.cs:84-125`）
- `ProxyEmitter.EmitClassConstructors` 不处理 `required`（`ProxyEmitter.cs:381-410`）

**适配方案**

1. `ConstructorNode` 增加 `SetsRequiredMembers` 标志
2. 检测实现类构造函数上的 `[SetsRequiredMembers]` 属性并转发
3. 代理构造函数上 emit `[SetsRequiredMembers]` 属性
4. `required` 属性的访问器通常是 `init`，需与 init 适配联动
5. SG 中构造函数签名需添加 `[SetsRequiredMembers]`

**适配难度**：⭐⭐⭐⭐

---

#### 3.1.4 record 类型 — C# 9.0

**问题描述**

Record 默认 `sealed`，且合成了 `Equals`、`GetHashCode`、`Deconstruct`、`<>Copy` 等成员。当前处理：

- Source Generator：拒绝 sealed 非抽象类型（`IsSealed && !IsAbstract` → ACSG003 诊断）
- DynamicProxy：**无 sealed 检查**，运行时崩溃

**影响范围**

- `AspectCoreProxyGenerator.cs:105` 拒绝 sealed 类型
- `ILEmitVisitor.cs` 无 sealed 类型检查
- Record 合成成员（`<>Copy`、`with` 表达式）无特殊处理
- `record struct`（值类型 record）无处理

**适配方案**

1. DynamicProxy 增加 sealed 类型检查（与 SG 对齐），提前报错而非运行时崩溃
2. 对于 `record class`（非 sealed 场景）：正确转发合成成员，`with` 表达式需要 `<>Copy` 方法被代理
3. 对于 `record struct`：值类型代理需要额外处理（装箱/拆箱）
4. 考虑支持 `record` 接口代理（record interface 不是 sealed）
5. Record 的 `init` 属性需与 init 适配联动

**适配难度**：⭐⭐⭐⭐

---

#### 3.1.5 ref struct — C# 7.2

**问题描述**

ref struct（如 `Span<T>`、`ReadOnlySpan<T>`）有 CLR 强制限制：

- 不能装箱
- 不能作为接口实现
- 不能作为类字段
- 不能用于 `async`/`iterator` 方法

DynamicProxy 会尝试在 IL 中创建 ref struct 的装箱操作，导致 `TypeLoadException`。

**影响范围**

- `ProxyTypeGenerator` 无 `IsByRefLike` 检查
- `AspectCoreProxyGenerator` 无 ref struct 拒绝逻辑
- 方法参数/返回值中的 ref struct（如 `Span<T>` 参数）可能被错误装箱

**适配方案**

1. 在 `ProxyTypeGenerator` / `AspectCoreProxyGenerator` 入口检查 `type.IsByRefLike`
2. 对 ref struct 类型直接拒绝并给出明确诊断信息
3. 对于方法参数/返回值中的 ref struct（如 `Span<T>` 参数），确保 IL 中不装箱，使用 `EmitLoadArg` 直接传递
4. 拒绝将 ref struct 作为代理目标类型（因为无法装箱到 `object`）

**适配难度**：⭐⭐⭐

---

#### 3.1.6 ref 返回方法 — C# 7.0

**问题描述**

当前只处理 `ref`/`out`/`in` **参数**，不处理 `ref` **返回值**。`ReturnKind` 枚举无 ref return 路径，`EmitReturnValue` 不处理按引用返回。

**影响范围**

- `ReturnKind` 枚举缺少 `Ref` 值
- `ILEmitVisitor.EmitReturnValue` 无 ref return 处理
- `AspectActivator` 缺少 `InvokeRef<T>` 方法
- `ProxyEmitter.EmitProxyInvokeBody` 无 ref return 代码生成

**适配方案**

1. 新增 `ReturnKind.Ref` 枚举值
2. `EmitReturnValue` 中处理 ref 返回：拦截器链的返回值需要是 `ref` 语义
3. `AspectActivator` 可能需要 `InvokeRef<T>` 方法，返回 `ref T`
4. IL 层面需要使用 `stind`/`ldind` 系列指令操作引用
5. SG 中生成 `ref` 返回方法的代理逻辑

**适配难度**：⭐⭐⭐⭐

### 3.2 P1 — 功能缺失但不导致崩溃

#### 3.2.1 主构造函数（Primary Constructors）— C# 12.0

**问题描述**

当前构造函数转发逻辑用 `GetParameters()` 通用处理，不感知主构造函数语义。对于 class 的主构造函数，参数需要被 `base()` 正确转发；对于 record 的主构造函数，还涉及 `init` 属性和 `with` 表达式。

**影响范围**

- `ClassProxyCtorFromBase` 虽然转发参数，但不处理主构造函数参数被属性捕获的情况
- `ProxyEmitter.EmitClassConstructors` 使用 `: base(...)` 转发，但不感知主构造函数

**适配方案**

1. 检测主构造函数（`ConstructorInfo` 上的 `[PrimaryConstructorParameters]` 或类型特征）
2. 确保主构造函数参数被正确转发到 `base()`
3. 对于 record 主构造函数，处理 `init` 属性和 `with` 表达式的 `<>Copy`
4. 测试主构造函数参数与属性初始化器的交互

**适配难度**：⭐⭐⭐

---

#### 3.2.2 params 集合（params IEnumerable\<T\>）— C# 13.0

**问题描述**

无处理。`params` 集合参数在方法签名中有 `[ParamCollection]` 属性，代理方法需要正确转发该属性并保持 `params` 语义。

**影响范围**

- `ParameterNodeFactory` 不检查 `[ParamCollection]` 属性
- IL 发射不转发 `[ParamCollection]` 属性
- SG 不生成 `params` 修饰符

**适配方案**

1. `ParameterNode` 增加 `IsParamsCollection` 标志
2. 检测 `[ParamCollection]` 属性并转发
3. IL 中使用 `SetCustomAttribute` 转发 `[ParamCollection]`
4. SG 中生成 `params` 修饰符

**适配难度**：⭐⭐

---

#### 3.2.3 ref 字段 / ref struct 字段 — C# 11.0

**问题描述**

ref struct 中的 `ref` 字段和 `scoped` 关键字影响参数传递。当前不处理 `scoped` 修饰符，可能导致在 `ref struct` 上下文中生成无效 IL。

**影响范围**

- 参数转发不保持 `scoped` 修饰符
- ref 字段在代理类型中无法复制

**适配方案**

1. 检测 `scoped` 参数修饰符（`ScopedRef` 属性）
2. 在 IL 和 SG 中转发 `scoped` 语义
3. 对于 ref struct 类型，拒绝作为代理目标（与 3.1.5 联动）

**适配难度**：⭐⭐⭐

---

#### 3.2.4 部分属性（Partial Properties）— C# 13.0

**问题描述**

新的成员类型。当前 `PropertyNode` 不区分 partial 属性，SG 的 `EmitAccessorBody` 可能需要为 partial 属性生成不同的访问器体。

**影响范围**

- `PropertyNode` 无 partial 属性标记
- SG 不区分 partial 属性的访问器生成

**适配方案**

1. `PropertyNode` 增加 `IsPartial` 标志
2. SG 中为 partial 属性生成正确的访问器声明
3. 确保 partial 属性的实现部分被正确代理

**适配难度**：⭐⭐

---

#### 3.2.5 插值字符串处理器（Interpolated String Handlers）— C# 10.0

**问题描述**

新的 struct 类型（如 `DefaultInterpolatedStringHandler`、自定义 handler）作为返回值或参数时，需要正确识别其类型并走 `Sync` 路径。当前无特殊问题但也无测试覆盖。

**影响范围**

- 自定义插值处理器类型作为参数/返回值时的类型识别
- 无测试覆盖

**适配方案**

1. 确认 `Sync` 返回路径能正确处理插值处理器 struct 类型
2. 添加测试用例覆盖插值处理器参数和返回值

**适配难度**：⭐

---

#### 3.2.6 System.Index / System.Range — C# 8.0

**问题描述**

作为参数或返回值时是普通 struct，理论上 `Sync` 路径可覆盖，但无测试覆盖。`Range`/`Index` 的 `^` 和 `..` 语法编译为方法调用，不影响 IL。

**影响范围**

- 无测试覆盖 `Index`/`Range` 参数和返回值

**适配方案**

1. 确认 `Sync` 路径正确处理 `Index`/`Range` struct
2. 添加测试用例

**适配难度**：⭐

## 6. 关键适配点详解

### 6.1 IAsyncEnumerable\<T\> / IAsyncDisposable

**核心挑战**：异步流的 `yield return` 语义要求拦截器链在每次 `MoveNextAsync` 时执行，而非一次性拦截整个方法调用。

**实现要点**：

- `AspectActivator` 需要新增 `InvokeAsyncEnumerable<T>` 方法，返回 `IAsyncEnumerable<T>`
- 使用 `AsyncIteratorMethodBuilder` 创建异步迭代器状态机
- 拦截器链需要支持延迟执行：在 `MoveNextAsync` 调用时执行拦截器逻辑
- IL 发射需要生成 `IAsyncEnumerable<T>` 的状态机结构（`<>d__` 嵌套类型）

**伪代码**：

```csharp
// AspectActivator 新增方法
public IAsyncEnumerable<T> InvokeAsyncEnumerable<T>(AspectActivatorContext context)
{
    // 在每次 MoveNextAsync 时执行拦截器链
    return new AsyncEnumerableProxy<T>(context, _aspectBuilderFactory);
}
```

### 6.2 init-only 设置器

**核心挑战**：`init` 访问器在 IL 中通过 `MethodImplAttributes.InitOnly` 标志标识，代理必须保持此标志。

**实现要点**：

- `PropertyNode` 增加 `IsInitOnly` 字段，从 `MethodBase.IsInitOnly` 读取
- IL 路径：`ILEmitVisitor.VisitProperty` 中对 init 访问器调用 `methodBuilder.SetImplementationFlags(MethodImplAttributes.InitOnly)`
- SG 路径：`ProxyEmitter` 生成 `init` 关键字
- 构造函数中 init 调用前确保 `this` 已加载到栈上

### 6.3 required 成员

**核心挑战**：`required` 属性带有 `[RequiredMember]` 属性，构造函数需要 `[SetsRequiredMembers]` 才能在对象初始化器中设置。

**实现要点**：

- 检测实现类构造函数上的 `[SetsRequiredMembers]` 属性
- 代理构造函数转发 `[SetsRequiredMembers]`
- `required` 属性通常与 `init` 联动，需一并适配

### 6.4 record 类型

**核心挑战**：Record 默认为 sealed，合成成员（`<>Copy`、`Equals`、`GetHashCode`、`Deconstruct`）需要正确代理。

**实现要点**：

- DynamicProxy 增加 sealed 类型检查，提前报错
- 非 sealed record class：转发合成成员，`with` 表达式需要 `<>Copy` 被代理
- record struct：值类型代理需处理装箱/拆箱
- record interface：可作为接口代理目标

### 6.5 ref struct

**核心挑战**：ref struct 不能装箱、不能作为类字段，DynamicProxy 会在 IL 中触发 `TypeLoadException`。

**实现要点**：

- 入口检查 `type.IsByRefLike`，拒绝代理 ref struct 类型
- 方法参数中的 ref struct（如 `Span<T>`）确保不装箱，直接 `EmitLoadArg`
- 给出明确的诊断信息

### 6.6 ref 返回方法

**核心挑战**：`ref` 返回方法返回变量引用而非值，拦截器链的返回值也必须是 `ref` 语义。

**实现要点**：

- 新增 `ReturnKind.Ref` 枚举值
- `AspectActivator` 可能需要 `ref T InvokeRef<T>(...)` 方法
- IL 中使用 `ldind`/`stind` 系列指令操作引用
- SG 中生成 `ref` 返回方法的代理逻辑

## 7. 总结

| 类别 | 数量 | 关键项 |
|------|------|--------|
| ✅ 已适配 | 6 类 | 协变返回、async/await、泛型、Nullable\<T\>、ref/out/in 参数、ValueTuple |
| 🔴 P0 需适配 | 6 项 | IAsyncEnumerable、init、required、record、ref struct、ref return |
| 🟡 P1 需适配 | 6 项 | 主构造函数、params 集合、ref 字段、部分属性、插值处理器、Index/Range |
| 🟢 P2 低优先 | 6 项 | NRT、泛型属性、CallerArgumentExpression 等注解类 |
| ⚪ 无需适配 | 30+ 项 | 纯语法糖，编译为标准 IL |

**最大风险点**：`IAsyncEnumerable<T>`（异步流越来越普及）、`init` + `required`（C# 11+ 推荐模式）、`record` 类型（现代 C# 常用类型）。这三项应作为最高优先级适配目标。
